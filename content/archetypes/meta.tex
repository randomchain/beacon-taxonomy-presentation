\note{
    \begin{itemize}
				\item $t_-1$: Data reception starts.
				\begin{itemize}
					\item Concatenated in order of arrival.
					\item Any user should try to tweet as close to $t_0$ as possible to make sure $t_0$ is the deadline.
				\end{itemize}
				\item $t_0$: Data reception stops
				\begin{itemize}
					\item After a small epsilon (less than a second), the concatenated tweets and a commitment to salt published.
					\item Sloth is started on tweets concatenated with salt.
					\item Sloth is just a hash function that is very slow.
				\end{itemize}
				\item $t_1$ The result of the sloth is the last hash.
				\begin{itemize}
					\item the salt is published
					\item and the witness is a shortcut to verification.
				\end{itemize}
        \item Which attacks?
        \item Needs some way of protecting against attacks
    \end{itemize}

\note{
    \begin{itemize}
				\item Specialized MPC
				\begin{itemize}
					\item Complex mechanisms, security governed by heavy math and protocols
					\item Scalability issues that needs to be solved
					\item Must participate by ``being there'' and do work
				\end{itemize}
        \item Transparent Authority
				\begin{itemize}
					\item (where s-mpc governed by math) Governed by simpler things such as the logic behind slow hash functions in Random Zoo.
				\end{itemize}
    \end{itemize}
}

\begin{frame}{Archetype Details}
    \begin{itemize}
			\item Autocratic Collectors are obviously not ideal in our setting.
			\item S-MPC suffers from scalability issues and all participants is expected to perform work and ``be there'' for a long time. Probably best suited for permissioned settings.
			\item Transparent Authority is a middle point. Full transparency, but significantly simpler construction than S-MPC.
		\end{itemize}
\end{frame}
